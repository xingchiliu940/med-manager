## 2. 背景与目标

### 2.1 业务背景（摘要）

慢性病患者在问诊、复诊、购药、用药、续方链条上体验割裂；本产品以单页 Web 应用形态聚合**用药管理 + 问诊演示闭环**，首版以**本地单机、单用户**、数据持久化于浏览器为主。

### 2.2 技术目标

| 维度 | 目标 |
|------|------|
| 形态 | SPA，底部五 Tab 主导航，移动优先、桌面端最大内容宽度 640px 居中 |
| 数据 | MVP 全量业务数据本地化；预留后续 API 层抽象 |
| 体验 | 首屏与性能口径对齐 PRD 第 6.1 节（LCP/FCP、Fast 3G Profile） |
| 合规展示 | 免责声明、模拟数据说明按 PRD 第 6 节固定位置实现 |

### 2.3 明确不做（MVP 边界）

- 多用户/多档案、跨设备云同步、真实支付与处方合法性链路（演示占位）。
- 独立主导航「提醒中心」Tab（逻辑收敛至首页 + 个人中心子页 + 业务内上下文）。
- MVP 内「日历导出」无入口、不测（PRD 第 11 节）。
- 跨标签页实时协同：以最后写入本地存储为准；可选 `storage` 事件提示（README 建议单标签）。
- **P-05 防重复囤药**（PRD P2）：MVP 不做交互；二迭单独立项评审。
- **文案与语言**：MVP 界面与 Toast **硬编码简体中文**，不引入 i18n 框架。

## 4. 技术选型建议

| 类别 | 建议方案 | 说明 |
|------|----------|------|
| 框架 | React 18+ 或 Vue 3+（团队栈二选一） | 组件化与生态成熟；若 PRD 已锁定栈则服从仓库 |
| 语言 | TypeScript | 实体与枚举强类型，减少业务 bug |
| 构建 | Vite | 冷启动与 HMR 友好，利于 LCP 优化 |
| 路由 | React Router / Vue Router | 五 Tab + 二级子路由；保持「页面层级」≤ PRD 2.3.1 |
| 状态 | 轻量：Zustand / Pinia 或 Context+reducer | 避免过重；领域逻辑可抽离为纯函数模块 |
| 样式 | CSS 变量（对齐 PRD 第 4.2、7 节 token）+ 可选 Uno/Tailwind 仅作工具类 | 色值与 `--tag-*` 与 PRD 一致 |
| 持久化 | `localStorage` + 单一 root key | key 命名统一，便于清空与导出（二迭） |
| 日期 | 原生 `Date` + 明确「业务日」封装 | 一律设备本地时区；不处理跨时区同步 |
| 通知 | `Notification` + 站内横幅/红点 | 权限与勿扰逻辑见第 8 节 |
| 兼容 | ES2020 子集 + **附录 A** polyfill 上限 | 不测 IE11 |

**Q-01 配置**：科室等使用**构建时打包**的静态 JSON 或 TS 常量模块；不设远程配置层（未来可抽象 `ConfigProvider`）。

## 6. 数据模型与存储设计

### 6.1 持久化根结构（建议）

```typescript
// 说明：以下为类型示意，字段名可按项目规范微调，语义须与 PRD 第 8 章一致
interface AppPersistRoot {
  schemaVersion: number;           // 见 12.1 迁移
  userProfile: UserProfile;
  drugMasters: DrugMaster[];       // 药品主数据
  stockBatches: StockBatch[];      // 或嵌套在 DrugMaster 内，二选一须统一扣减入口
  medicationPlans: MedicationPlan[]; // 单条可含 sourcePrescriptionId 等溯源字段，见 7.9
  doseRecords: DoseRecord[];       // 服药记录（应服实例）
  consultations: ConsultationOrder[];
  prescriptions: Prescription[];
  followUps: FollowUpEvent[];
  reminders: ReminderFeedItem[];   // 未读/已读或统一时间线，见 8.2
  settings: AppSettings;           // 阈值、勿扰、R-02 开关、字号等
  meta: { disclaimerAcceptedAt?: string; lastLargeTimeJumpHintAt?: string };
}
```

### 6.2 核心实体字段（与 PRD 8.2、8.5 对齐要点）

- **药品主数据**：名称、规格、库存单位、总库存（可派生）、低库存阈值、规范化合并键（存盘可缓存）。
- **库存批次**：药品主数据 ID、数量、入库时间、来源枚举（手动 / 处方导入 / 购药确认）、可选效期。
- **用药计划**：关联药品主数据 ID、每次剂量数值、剂量单位、频次类型与参数（见 7.2）、时间点数组、疗程起止、启用/停用、`createdAt`（用于同日多计划扣减顺序）。
- **服药记录（应服实例）**：计划 ID、应服日期（本地日历日）、时段标识、状态（待服/已服/漏服/补服）、操作时间戳；补服需记录「针对的应服实例」避免重复扣库存。
- **问诊订单**：模式、科室、医生、预约时间、Q-06 状态、摘要等。
- **处方**：明细行、有效期、关联问诊 ID、来源枚举（`image_consult`图文问诊生成 / `renewal_consult`续方问诊生成 / `manual_import`手动录入）。
- **复诊事件**：时间、类型、医生、科室、周期规则（F-02 二迭可加重）。

### 6.2.1 ReminderFeedItem 结构定义

提醒系统的核心数据结构，用于持久化站内提醒时间线：

```typescript
interface ReminderFeedItem {
  id: string;                    // UUID，唯一标识
  businessKey: string;           // 业务主键（如 prescriptionId、planId），用于 PRD 5.6 去重
  type: ReminderType;            // 枚举：'dose' | 'stock_low' | 'rx_expiring' | 'rx_expired' | 'renewal' | 'follow_up' | 'consult_booked'
  title: string;                 // 展示标题
  body: string;                  // 展示正文（多行用 \n 分隔）
  read: boolean;                 // 已读状态；未读集合内按 businessKey+type 去重
  createdAt: string;             // ISO 8601，用于排序（倒序）
  triggeredAt: string;           // 业务事件触发时间（可能早于 createdAt）
  relatedId?: string;            // 关联业务实体 ID（处方/计划等），用于跳转
}

type ReminderType = 'dose' | 'stock_low' | 'rx_expiring' | 'rx_expired' | 'renewal' | 'follow_up' | 'consult_booked';
```

**去重逻辑**：未读集合内，相同 `businessKey + type` 组合只保留 1 条（后来的覆盖前者）；已读后再次触发可新建记录。

### 6.3 药品合并键规范化（PRD 8.1）

实现为纯函数管道：

1. Unicode NFC  
2. 全角 → 半角（数字与常见单位）  
3. 连续空白折叠为单空格  
4. 英文小写  
5. 剂量单位映射表（毫克/mg → `mg` 等）  
6. 规格串仅空白与全半角规范化，不做医学等价推断  

**无法映射单位**：MVP **采用阻断保存并提示**（要求用户统一单位或修正录入）；不自动拆成两条主数据，避免静默数据分叉。

**「作为新品种」高级拆分**（PRD 8.1）：MVP **不提供**「强制不合并」入口；二迭再评估是否增加显式拆条。

### 6.4 扣减与批次（PRD 8.1、8.2、12.3）

- 单次打卡扣减量 = 每次剂量换算到**库存单位**后的数量；录入阶段强制口径一致或提供换算规则。  
- 批次扣减：**FIFO**（入库时间最早且剩余 > 0）。  
- 同一药品多条启用计划同时应服：先按计划 `createdAt` 升序，再 FIFO 批次。  
- **首页 H-01 同药多行合并打卡**（PRD 12.3）：MVP **不提供**「用户选择扣哪条计划」；从合并卡片触发已服/补服时，**严格按 `createdAt` 升序**依次扣减对应计划的单次用量与库存；若中途库存不足，按 12.3 禁止后续条目的已服操作并 Toast。  
- **合并卡片打卡后 UI 刷新规则**：单次打卡确认后，卡片**整体刷新**（不拆分为多张卡），继续以合并态展示剩余待服条目；若该药当日所有计划均已打卡，则卡片从首页 H-01 移除或折叠为「已全部服用」完成态。库存不足中断时，**已成功扣减的前几条不回滚**（保持已服状态），仅对当前及后续条目禁用操作并 Toast 说明。
- 库存不足或 0：**禁止**已服/补服（主按钮 disabled + 固定 Toast 文案）。

### 6.5 近效期批次预警（PRD 8.2、10.2）

- **常量**：默认 **90 天**（与附录 B「近效期预警」一致）；写入 `settings` 或代码常量，二迭可下放设置。  
- **判定**：批次填写效期后，若 `效期日期 − 今日（本地日历日）≤ 90` 且 `> 0`，在**库存详情**与**首页 H-04 摘要**展示弱警示标签「近效期」。  
- **与库存不足同时命中**：卡片**标题优先「库存不足」**；副标题或次行展示「近效期：yyyy-mm-dd」。  
- **浏览器通知**：与库存等同日多条时仍遵守第 8 章合成规则，**最多 1 条**，正文可分两行分别描述临期与库存（与 PRD 8.2 末段一致）。

## 7. 核心业务逻辑与算法

### 7.1 每日应服清单生成（PRD 8.4）

- 支持类型：每日 N 次（N∈[1,6]）、等间隔映射为每日 N 次、每日 1 次。  
- 不支持类型：表单禁用 + 灰色说明；提交阻断并 Toast。  
- 输入：启用中的计划 + 本地日历日 D；输出：当日应服实例列表。

**生成策略收口（MVP 选定：懒生成 + 幂等）**

MVP 采用**懒生成**策略：不预先写入未来的应服记录，而是在读取时按计划参数动态计算当日/历史日的应服实例列表；用户打卡或补服时才将该条实例写入 `doseRecords` 持久化。

- **幂等保证**：同一计划 + 同一日期 + 同一时段最多一条 `doseRecord`（主键三元组），写入前先查重。  
- **历史补记**：任意历史日懒生成应服实例，与持久化的 `doseRecords` 合并后展示，状态以持久化记录为准。  
- **疗程结束/处方过期**：懒生成时检查计划 `endDate` 与处方 `expireDate`，超出范围的日期不生成应服实例。  
- **存储体积**：只有实际发生打卡/漏服标记的记录落盘，历史未操作的日期不产生记录。

### 7.2 疗程结束（PRD M-01）

- 到达疗程结束日 **24:00（本地）** 后计划自动停用，不再生成新待服；已生成未打卡按漏服规则归档。  
- 可选：列表灰态 +「按原方案延长」二次确认 +「复诊开方」跳转。

### 7.3 处方有效期与计划（PRD 12.4）

- 处方过期日次日 00:00：关联计划自动停用，不再生成待服；仅历史查看，不可再打卡。

### 7.4 预约状态机 Q-06

| 转移 | 条件 |
|------|------|
| 已预约 → 待就诊 | 预约日 00:00（本地定时任务或每次启动/焦点时 reconcile） |
| 待就诊 → 已完成 | 用户「标记已完成」/「一键完成就诊」 |
| 待就诊 → 已完成（自动） | 超过预约日 +1 天 24:00 仍未完成 → 已完成（展示「已逾期自动完成」类文案） |
| 已取消 | 用户取消（不回收号源、无排班冲突校验） |

爽约不按「已取消」处理；H-03 在超时前展示待就诊。

### 7.5 P-06「用本处方更新用药方案」

1. **入口**：仅处方详情页底部主按钮。  
2. **步骤**：① 预览差异（只读表，名称匹配用 8.1 规范化键）；② 选择范围（默认全量，高级按行勾选）；③ 二次确认固定文案。  
3. **执行**：对选中范围：旧处方关联计划中同通用名+规格条目**默认停用**；新明细生成新计划条目；历史打卡保留。  
4. **撤销**：当次会话 Toast「已更新」+ 5 秒内「撤销」；恢复操作前计划启用快照（若实现困难按 PRD 降级并在 README/评审备案）。  
   - **快照最小结构（建议）**：`{ planId, wasEnabled: boolean }[]`，仅覆盖本次 P-06 影响到的计划；撤销时逐条写回 `wasEnabled`，不触碰未选中行。内存持有即可，**不落盘**；刷新或关标签即失效。  
   - **降级**：无法实现原子恢复时，Toast 引导用户至用药计划手动启用，并在评审记录中备案。  
5. **离开应用或超时**：不提供撤销。

### 7.6 服药统计与补记（PRD 8.3、8.3.1）

- 分母/分子、漏服、补服、跨日补服、库存扣减次数与记录归属日**严格按 PRD 表格实现**。  
- 补记入口：**用药计划** → 日历/按日列表 → 选历史日 → 对漏服条目补服。  
- 宽限期：默认当日 24:00 前补服仍计该计划日依从；超窗策略按 PRD MVP 建议。

### 7.7 复诊与线上面诊同日（PRD 12.5）

- 数据层两条保留；首页 H-03 **合并一张卡片**，标题「今日就诊安排」，正文两行；点击进**半屏聚合**：建议统一为 **底部抽屉（Bottom Sheet）** 或全屏次级页二选一，全站一致；内文可展开两条记录详情链接。

### 7.8 购药入库未同步计划（PRD 12.6）

- 仅增加库存批次；H-01 不出药；H-04 仍可按阈值展示；首页**常驻轻提示条**（可关闭一次）引导 M-01 预填药名。

### 7.9 处方导入用药计划（M-02，PRD 5.4）

1. **入口**：处方详情页「加入用药计划」或等价主操作；与 P-06「更新方案」区分——M-02 为**新增可编辑计划条目**，不自动停用旧计划。  
2. **数据流**：读取处方明细行 → 对每行用 **6.3 合并键** 匹配或新建 `DrugMaster`（入库来源标记为处方导入时，按 PRD 8.1 累加/新建批次策略）→ 生成 `MedicationPlan` 草稿（频次、时间点、疗程默认来自明细或合理默认，**允许用户保存前编辑**）。  
3. **关联**：`MedicationPlan` 记录 `sourcePrescriptionId`（及可选行号）便于 P-06 与「与处方不一致」弱标签（12.4）溯源。  
4. **校验**：剂量单位与库存单位不一致时，与 M-01 相同阻断或换算规则。

### 7.10 跨日服药与操作日归因（PRD 3.3、8.3）

- **口径**：用户在 **次日 0:00 之后** 对「昨日」应服条目打卡或补记时，**统计与库存扣减归属以用户点击确认时的本地日历日为准**（与 PRD 8.3「补服操作日」列一致）。

- **H-02**：跨日补服时分子仅计操作日，原计划日不因补服操作增加分子（与 PRD 8.3.1 一致）。

## 8. 提醒、通知与站内消息

### 8.1 类型与互斥（PRD 5.6、R-01）

- 类型：问诊预约、复诊到期、服药、库存不足、处方到期、续方建议等。  
- **同一处方同一自然日**：续方 + 临期合并为一条卡片/通知，不重复弹窗。  
- **续方与临期钳制**：生成提醒前执行 **附录 C** 算法，避免孤立续方空提醒。

### 8.2 渠道与优先级（PRD F-02、5.6）

- **站内横幅**：多条时按 **处方/复诊 > 库存 > 服药** 合并为一条多行正文；不叠多个模态。  
- **浏览器通知**：需 R-02 总开关 + 子开关；同日多条最多 1 条合成（标题如「用药提醒」）。  
- **勿扰 R-03**：站内复诊提醒延迟到勿扰结束首次展示（不丢）；浏览器通知勿扰内抑制，结束后合并补发 1 条；服药同理。  
- **H-02 分母**：以计划日期为准，不因提醒展示推迟而改变。

### 8.3 无系统通知权限（PRD 5.6）

- 红点/角标：个人中心「首页显示提醒角标」默认开启；关闭则首页 Tab 无数字角标，横幅与常驻条保留。  
- 冷启动/回前台：未读提醒可顶部横幅或抽屉批量展示。  
- 首次拒绝权限：Toast 说明不授权仍可使用。

### 8.4 批量列表「最近 10 条」（PRD 5.6）

- 排序：事件产生时间倒序。  
- 去重：同一业务主键 + 类型在未读集合内只保留 1 条。  
- 「查看全部」→ 个人中心提醒历史（不去重、时间倒序）。

### 8.5 实现要点

- **Reconcile 主策略**：以 `document.visibilitychange`（页面从隐藏恢复可见）作为**主触发器**强制执行一次 Q-06 状态推进、处方过期检查、日界切换；`setInterval`（建议 60s）作为**补充兜底**。移动端浏览器后台会 throttle/suspend 定时器，故 visibilitychange 优先级更高，不可省略。  
- `Notification` 请求时机：用户进入「提醒与通知」设置页再请求，避免盲弹。  
- 服药提前量：设置项 5/10/15/30 分钟，与本地闹钟式提醒（若仅依赖页面内定时器，需说明**页关则无推送**——与 PRD「合理预期」文案一致）。

### 8.6 续方与临期钳制算法（与附录 C 一致，实现摘要）

全站统一 **`R`（处方剩余完整自然日）**：从**明日**起算至**失效日前一日** inclusive 的天数，**不含失效日当天**（与 PRD 10.2 建议一致）。设用户配置的续方提前天数为 **`U`**（来自设置，附录 B）。

1. 若 **`R = 0`**（已过期或当日即失效后的过期链路）：走「处方已过期」提醒，不单独发续方类孤立提醒。  
2. 若 **`R > 0` 且 `U ≥ R`**：**禁止**在「失效日 − `U`」单独产生**仅续方类型**的提醒；改为在进入**处方临期窗口**（用户配置的临期天数 `L`，见附录 B）时发 **一条合并卡片**（续方建议 + 临期信息，与 PRD 5.6 节文案一致）。  
3. 若 **`U < R`**：续方提醒按「失效前 `U` 天」规则独立生成；若与临期同日仍合并为一条（8.1 互斥）。

**验收锚点用例**（照 PRD 10.2）：剩余 `R = 5` 天、`U = 14` → 不出现「只剩 5 天却提前 14 天续方」的空洞提醒；进入临期窗后 **1 条合并**。

## 附录 C：`R` / `U` 续方与临期钳制（PRD 10.2 可实现版）

### C.1 符号

- **`R`**：处方当前剩余 **完整自然日** 数。全站约定：**不含失效日当天**（与失效日当日是否仍「可用」的展示文案配合，由产品一句口径统一；计数上 `R` 表示「失效日之前还剩几天完整日」）。实现建议：`R = max(0, 以本地日历日计算的失效日前一日 − 今日)` 的日差，或等价按 PRD 用例表校准。  
- **`U`**：用户设置的续方提前天数（附录 B）。  
- **`L`**：处方临期提醒天数（附录 B）。

### C.2 生成规则（提醒服务）

1. **`R = 0`**：仅走「处方已过期 / 不可用」链路，**不**单独生成「仅续方建议」类孤立提醒。  
2. **`R > 0` 且 `U ≥ R`**：**不**在「失效日 − `U`」生成仅续方类型提醒；待进入临期窗（距失效 ≤ `L` 天，口径与 PRD 5.6 一致）时生成 **一条合并提醒**（续方建议 + 临期信息）。  
3. **`U < R`**：按「失效前 `U` 天」生成续方类提醒；若与临期规则同日触发，仍 **合并为一条**（PRD 5.6 R-01 互斥）。

### C.3 验收用例（照抄自测）

- 剩余 **`R = 5`**、**`U = 14`**：不出现「提前 14 天续方」的空洞单独提醒；进入 **`L`** 临期窗后 **1 条合并**。
