## 2. 产品概述

### 2.1 产品背景

慢性病患者在**线上问诊 → 复诊 → 购药 → 用药 → 续方**链条上体验割裂，常见痛点包括：

- 忘记按时服药、漏服、错服  
- 重复囤药、库存与用量不同步  
- 错过复诊、无处方续方规划  
- 问诊入口分散、历史记录难找  

本产品以**一站式闭环**为目标，在 Web 端聚合用药管理与问诊相关能力，降低认知与操作负担。

### 2.2 产品定位

打通 **在线问诊 → 复诊预约 → 处方留存 → 一键购药 → 定时用药提醒 → 药品库存管理 → 到期续方提醒** 的完整闭环，做慢病患者专属的**用药与问诊全流程管家**。

### 2.3 产品目标（可量化方向）

| 目标维度 | 描述 |
|----------|------|
| 体验 | 核心任务（今日服药、发起问诊、查看处方）**3 步内可达**（「步」的定义见下表 **2.3.1**，避免与研发/测试对「一步」扯皮） |
| 可信 | 状态清晰（待服药、库存不足、处方将过期等）一目了然 |
| 适老 | 大字号、高对比关键信息、减少专业术语堆砌 |

#### 2.3.1 「步」与页面层级口径（验收用）

- **「一步」**：计为 **一次用户点击** 触发的导航或提交，包括：**点击底部 Tab**、**点击列表项进入二级页**、**点击主按钮 / 文字链**（同一次连点只算 1 步）。  
- **不计步**：**滚动**、**在输入框内键入**、**系统手势返回**、**页面内 Tab 切换（同页签组件，未换路由）**。  
- **自动跳转**：问诊结束等 **系统/流程自动跳转** 下一页，计 **0 步**（但计入「页面层级」时仍算一次路由变化，需在测试用例中注明）。  
- **「页面层级」**（与「步」并列描述时）：**底部 Tab 切换次数 + 子页面栈深度（push）**；从首页 Tab 到目标内容区 **默认不超过 2 层 push**（不含弹层）。

**核心路径验收用例（研发自测 / 测试写用例时照抄）**

| 编号 | 场景 | 起点 | 终点（可见可操作） | 步数上限 | 层级上限 |
|------|------|------|-------------------|----------|----------|
| E-01 | 今日待服 | 冷启动（清空后首次进入或刷新后首页） | 首页 **H-01 今日待服药清单** 至少 1 条或空状态文案 | **≤ 3 步** | 首页为 Tab1，**0 次 Tab 切换** 即达 |
| E-02 | 发起问诊 | 冷启动 | 进入 **选科室** 或 **问诊表单** 首屏 | **≤ 4 步** | Tab 切到问诊 **1** + 列表 **1** + 进入表单 **1** = 典型 3 步，留 1 步冗余 |
| E-03 | 查看处方 | 冷启动 | **复诊处方** Tab 下列表首屏或详情 | **≤ 4 步** | Tab 到复诊处方 **1** + 至多 **2** 次点击进入详情 |

> 若某路径因合并 Tab 导致层级变化，在迭代说明中 **更新本表** 并同步测试用例。

### 2.4 角色与实现约束（实现侧）

- **角色设定**：资深全栈产品设计师 + 前端工程师，基于 Vibe Coding 产出可运行单页应用。  
- **技术约束**：逻辑清晰、组件化、数据可先本地存储（如 `localStorage`）演示完整流程，后续可再接真实问诊/支付接口。

## 3. 用户画像与场景

### 3.1 核心用户画像

| 画像 | 特征 | 核心诉求 |
|------|------|----------|
| 中老年慢病患者 | 可能视力下降、对复杂导航不耐受 | 少点几下、字大、提醒可靠 |
| 家属照护者 | 代为管理父母用药与复诊 | 统一视图、记录可查、提醒可配置 |

### 3.2 典型用户故事（节选）

1. **作为** 高血压患者，**我希望** 每天打开首页就能看到今天要吃的药和是否已吃，**以便** 不漏服。  
2. **作为** 需要续方的用户，**我希望** 处方快过期时有明显提示并能一键进入复诊/问诊，**以便** 不断药。  
3. **作为** 用户，**我希望** 问诊记录和处方能存档回看，**以便** 复诊时给医生看或自己核对。

### 3.3 首版用户与设备边界（评审收口 U-01 / U-02）

- **多用户与家属照护**：首版 **仅支持本地单机、单用户档案**，不设多档案切换、家属账号绑定或数据共享。文案中「家属照护」指 **同一设备上由家属代为操作** 的使用方式，非远程协同。后续若做多用户，需另立迭代与数据隔离方案。
- **时区与跨日服药**：首版 **以用户设备本地时区与本地日历日（00:00～23:59）** 作为「今日服药清单」「当日完成率」「服药记录日期」的唯一口径；**换时区出行、跨时区同步** 不作为 MVP 目标，可在帮助文案中说明「出行时请仍以当地时间核对服药时间」。
- **跨日服药**：若医嘱时间为深夜（如 23:00）而用户在次日 0:00 后补记，**以用户点击打卡时的本地日期** 归入该日统计（产品可提示「您正在记录为【昨天/今天】的服药」以降低误操作）。

### 4.2 全局状态标签（可视化）

建议统一语义与颜色；实现时 **色值须与下表 CSS 变量（token）一致**，避免第 7 节与状态语义漂移。

| 状态文案 | 语义说明 | 建议展示色（对应第 7 节） | CSS 变量名（建议） |

| 已预约 | 问诊/复诊已约未发生 | 同上或次文案 `#5C6B7A` + 浅底 `#D9ECF7` | `--tag-booked` |
| 待服药 | 当日计划内尚未打卡 | 主色浅底 `#D9ECF7` + 主文案 `#2D3A45` | `--tag-due` |

| 漏服 | 超过宽限期仍未服且未补服 | 弱警示字 `#A67C00` + 底 `#FDF6E3` | `--tag-missed` |
| 补服已记 | 漏服后用户标记补服（可与「已服」同色系，文案区分） | 点缀绿或略深绿字 | `--tag-makeup` |

> 无障碍：状态除颜色外须有 **文案 + 图标** 之一或并存（见第 6 节）。

| Q-02 | 医生列表与简介 | 姓名、职称、擅长、可预约时段展示 | P0 |

| Q-04 | 问诊记录存档 | 列表 + 详情，历史会话可回看（演示可用本地 Mock） | P0 |

|---------|--------|------|--------|

| 需求 ID | 功能点 | 说明 | 优先级 |
|---------|--------|------|--------|
| P-01 | 处方统一管理 | 列表展示来源（问诊/导入）、药品明细、开具日期 | P0 |

| P-03 | 购药跳转 | 「一键购药」跳转合规渠道（演示可为外链占位 + 免责声明） | P1 |
| P-04 | 关联入库 | 购药确认后将药品写入库存台账，并可选择同步到用药计划。**MVP 无外部平台回调**：用户点击「确认已收到货/完成购药」即视为一次入库来源；后续对接真实电商/物流时，本入口改为「同步订单状态」占位，不改变首版演示逻辑。**未加入用药计划时** 的首页引导见 **第 12.6 节** | P0 |

| P-05 | 防重复囤药 | 同通用名+规格在库存充足时二次购药二次确认或弱化推荐 | P2 |

**P-06 交互与入口（评审收口：避免研发追问形态）**

- **入口**：**处方详情页** 底部主按钮 **「用本处方更新用药方案」**；不设独立一级「向导」Tab。点击后进入 **全屏或模态分步**：① **预览差异**（旧计划 vs 新处方：药品增删、剂量/频次变化，只读对比表）；② **选择范围**——默认 **全量药品** 参与更新，高级展开为 **按行勾选**（**允许只覆盖部分药品**；未勾选行保持原启用计划不变）；③ **二次确认**，主文案固定为：「将停用所选药品的旧处方关联计划并生成新计划，历史打卡仍保留。是否继续？」  
- **误触与撤销**：确认执行后 **当次会话内** 提供 Toast **「已更新」+ 操作「撤销」**（**5 秒内** 有效），撤销则恢复本次操作前计划启用快照（演示版实现若困难可降级为「请到用药计划手动启用」提示，但须在技术评审备案）。**离开应用或超时后** 不提供撤销，避免本地快照复杂度过高。  
- **与合并规则关系**：预览差异表中的名称须经 **第 8.1 节规范化键** 匹配；不匹配项视为新药行。

### 5.6 智能提醒中心

| 需求 ID | 功能点 | 说明 | 优先级 |

**续方与处方到期（R-01 互斥策略，评审 R-01）**

- **处方到期提醒**：处方失效日 **当天或已过期** 触发，文案侧重「已不可用，请尽快复诊开方」。
- **续方提醒**：在处方失效日 **之前** 触发，默认 **提前 N 天**（见第 10.2 节），文案侧重「建议安排复诊续方」。
- **同一处方同一自然日**：若同时命中两类规则，**只推送一条合并卡片**（标题如「处方即将到期」），正文同时含续方建议与到期日；**不重复弹窗/重复浏览器通知**。

**入口（评审 R-02）**

- **首版不设底部一级 Tab「提醒中心」**；提醒能力收敛为：① **首页** 聚合卡片（H-04、H-03、处方临期等）；② **个人中心 →「提醒与通知」子页**（总开关、各类型子开关、勿扰、提前量）；③ 各业务页内 **上下文入口**（如用药计划页调整服药提醒）。附录「八大模块」中的提醒中心与此一致，视为 **逻辑模块** 而非独立主导航。

**无系统通知权限时的默认行为（评审 R-03）**

- 不弹系统通知；**站内**：① 首页相关模块 **红点/角标**；② **用户下次打开或从后台切回应用时**，若有未读提醒，**顶部一次性横幅或抽屉列表** 批量展示（列表规则见下 **「批量提醒列表」**）；③ 关键状态（处方已过期、库存为 0）在对应 Tab **首屏常驻条** 展示。
- **红点/角标（评审收口）**：**配置入口** 在 **个人中心 → 提醒与通知** 子页，开关文案 **「首页显示提醒角标」**，**默认开启**。关闭后首页 Tab **不显示数字角标**，但 **横幅与常驻条仍保留**（避免用户完全无感知风险事件）。
- 首次拒绝权限时 Toast + 文案说明：**不授权仍可使用应用**，仅缺少锁屏外推送。

**批量提醒列表（「最近 10 条」规则，评审收口）**

- **排序**：按 **事件产生时间倒序**（最新在上）。  
- **去重**：同一 **业务主键 + 类型** 在 **未读集合内** 只保留 **1 条**（例：同一处方同日「续方+临期」已合并的卡片算 **1 条**）；已读后再次触发可新生成记录。  
- **「查看全部」**：进入 **个人中心 → 提醒与通知 → 提醒历史**（或与该子页同栈列表），历史列表 **不去重**、按时间倒序，便于审计。

**复诊提醒（F-02）与 R-02、站内横幅的关系**

- 见 **F-02** 与第 **5.6** 本节上文 **勿扰**、**合并优先级**；测试验收时 **对照 F-02 表格** 与 **本节批量列表** 联合用例。

**勿扰时段与「仅服药」提醒（R-03，与 F-02 对齐）**

- **浏览器通知**：勿扰内 **不响**；结束后若积压多条服药提醒，**合成 1 条** 系统通知（若仍受 R-02 总开关约束）。  
- **站内**：勿扰内 **不弹模态**；在 **勿扰结束时刻** 或 **用户主动打开应用时** 批量展示积压服药项（可与复诊合并为 **一条横幅多行**，优先级仍遵守 F-02：**处方/复诊 > 库存 > 服药** 在正文中的排序）。  
- **H-02 分母**：应服条目归属 **以计划日期为准**，**不因** 提醒展示推迟而改变。

**说明**：Web 端依赖用户授权与页面活跃策略，需在文案中说明「建议固定添加到主屏幕或保持常用浏览器打开」等合理预期。

### 6.1 性能目标（首屏定义与测量口径）

- **「首屏」**：指 **首路由可见内容区**（首页 Tab 内容）完成 **首次有意义绘制**：以 **LCP（Largest Contentful Paint）** 为主指标，**FCP** 为辅；**含首包 JS 解析与执行完成后** 能展示骨架屏或真实列表（二者取先达到「可交互」者）。**Web 字体**：若使用自定义字体，**计入首屏**；首版建议 **系统字体栈优先**，自定义字体为增强项并 **不阻塞** LCP 达标。  
- **冷启动**：无本站点有效缓存（或模拟清空缓存后）首次加载。**热启动**：已有缓存、二次进入同一会话。  
- **目标值（演示 / 开发自证，非运营商真实 4G 强约束）**：在 **本地或局域网预览** + **Chrome DevTools 网络限速：Fast 3G**（或 RTT 150ms、下行 1.6Mbps 档）Profile 下，冷启动 **LCP ≤ 3.0s**、**FCP ≤ 2.0s**（允许申报时附 Lighthouse 单次截图）。**不在 PRD 中要求** 在真实蜂窝弱网下复测（避免测试与研发对「4G」扯皮）；若商务需对外承诺，另附 **现网抽样** 报告。  
- **热启动**：二次进入首页 **LCP ≤ 1.5s**（同 Profile）。

  - 页面水平安全边距（内容与屏幕左右）：`16px`  
  - 标准卡片：圆角 `12px`；内边距 **`16px`**（四边一致）；卡片与卡片纵向间距 **`12px`**  
  - 强调 / 仪表盘摘要卡：内边距 **`20px`**；圆角仍为 `12px`  
  - 列表行卡片（药品、问诊记录等）：单行可视区域最小高度 **`72px`**（含内边距），左侧主文案区与右侧操作区间距不少于 `12px`  
  - 卡片描边：`1px solid #E2E8EF`；或与配色节一致的轻阴影二选一，全站统一即可  
  - 桌面端：主内容区最大宽度 **`640px`** 水平居中，两侧留白

  - 正文：`17px`（`1.0625rem`），行高 **`1.55`**，字重 `400`（全站正文下限不低于 **`16px`**）  
  - 页面大标题：`22px`（`1.375rem`），行高 **`1.35`**，字重 `600`  
  - 卡片标题 / 模块小标题：`18px`（`1.125rem`），行高 **`1.4`**，字重 `600`  
  - 辅助说明、表单标签、列表次要信息：`15px`（`0.9375rem`），行高 **`1.5`**；全站不得小于 **`14px`**  
  - 关键数字（如库存、倒计时、完成率）：`20px`～`22px`，行高 **`1.3`**，字重 `600`  
  - 底部 Tab 标签字：`15px`；选中态字重 `600`，未选中 `400`  
- **点击区域（适老）**：可点热区最小 **`44px × 44px`**；主按钮高度 **`48px`**、圆角 **`12px`**；纯图标按钮须扩透明热区至上述最小尺寸  
- **反馈**：操作成功 / 失败 Toast 文案用 **`15px`～`16px`**；加载骨架条高度建议 **`12px`**，圆角与卡片一致或 `8px`  
- **空状态**：每类列表无数据时给出一句人话说明 + 主操作引导；说明文案用 **`15px`～`17px`**
- **字号策略（评审 C-01，与 U-03 一致）**：  
  - **设计基准**：`html` 根字号默认 **`16px`**（`1rem = 16px`），第 7 节所列 `px` 为设计稿基准。  
  - **字号偏好（U-03）**：提供 **标准 / 大** 两档即可——**大** 档在基准上整体 **+2px**（且仍遵守正文不低于 16px、辅助不低于 14px 的下限），通过给根节点或内容容器加 `class` 覆盖 CSS 变量实现，**不依赖** 浏览器「仅字体缩放」破坏布局。  
  - **系统/浏览器缩放**：布局以 `rem` + 基准根字号为主；用户操作系统放大字体时，**优先保证不重叠、可滚动**，必要时次要信息折行，**不要求** 与第 7 节像素级完全一致。

## 8. 数据实体与业务规则（实现参考）

以下实体便于前后端对齐（首版可全本地化）。

### 8.1 实体关系（ER 概要，评审 D-01）

```text
用户档案 1 ── n 药品主数据（逻辑药品：通用名+规格+库存单位唯一）
药品主数据 1 ── n 库存批次（每次入库一条：来源、数量、可选效期）
药品主数据 1 ── n 用药计划（同一药品可多条计划，如早晚不同方案则拆计划）
用药计划 1 ── n 服药记录（每次应服一条：日期、时段、状态）
处方 1 ── n 处方明细行 ── 可生成/关联 用药计划（导入时复制剂量与频次）
问诊订单 0..1 ── 处方
```

- **「药品」「用药计划」「库存」关系**：**用药计划** 必须关联 **一条药品主数据**；**库存** 挂在药品主数据下，为 **多条批次**。**一条计划** 不强制唯一对应一条库存批次；扣减时见 **8.2**。
- **同名同规格多次录入**（手动 + 处方导入）：若 **规范化后的合并键**（见下）一致，**合并为同一药品主数据**，库存 **累加至默认批次** 或 **新建批次**（由入库来源决定）；若用户明确选择「作为新品种」则允许拆成第二条主数据（高级选项，MVP 可简化为仅合并）。  
- **合并键字符串规范化（评审收口，避免「5mg」与「5 毫克」各一条）**：匹配前对名称/规格/单位做 **Unicode NFC**、**全角→半角**（数字与常见单位符号）、**连续空白折叠为单空格**、**英文大小写统一小写**；**剂量单位映射表**（产品内置）：`毫克/mg/MG` → `mg`，`毫升/ml/ML` → `ml`，`片/片剂` → `片` 等；**无法映射** 则 **不参与自动合并**（视为两条主数据，或阻断保存并提示用户选单位）。**规格串**仅做空白与全半角规范化，**不做** 医学等价推断（避免误判）。  
- **扣减时选哪条库存**：默认 **先进先出（FIFO）**——按批次 **入库时间最早** 且 **剩余量 > 0** 的批次扣减；扣完再扣下一条。**同一药品多条启用计划、同一时刻多条应服**：打卡扣库存顺序为 **先按「计划创建时间」升序**，同一创建批次内再 **FIFO 批次**（与研发对齐：避免随机顺序）。用户可在库存详情中 **调整默认扣减批次**（P2，首版可不做仅 FIFO）。

### 8.2 字段补充与扣减公式（评审 D-02）

| 实体 | 必含字段（在 v1.0 基础上补充） |
|------|-------------------------------|
| 药品主数据 | 名称、规格、**库存单位**（盒/片/ml 等）、**当前总库存**（可由批次汇总）、**低库存阈值** |
| 用药计划 | 关联 `药品主数据 ID`、**每次剂量（数值）**、**剂量单位**、频次规则、时间点、疗程起止、启用/停用 |
| 库存批次 | `药品主数据 ID`、数量、入库时间、来源（手动/处方/购药确认）、可选 **批次效期**（用户录入或导入） |

**扣减公式**：每次用户对该计划的一次「应服次数」标记为 **已服** 或 **补服** 时，扣减量 = `每次剂量` 换算为 **库存单位** 下的消耗量（例：每次 2 片、库存按片计，则扣 2；若库存按盒、每盒 20 片，则内部换算为 2/20 盒或统一以片为库存单位，**产品须在录入时统一库存计量口径**）。

**近效期预警（批次级，评审收口）**：若批次填写了效期，且 **距效期 ≤ 90 天**（默认，可写入第 10.2 常量表），在 **库存详情与首页摘要** 展示 **「近效期」** 标签（弱警示色）；**与库存不足同时命中** 时：**卡片标题优先展示「库存不足」**，副标题或次行展示「近效期：yyyy-mm-dd」；**浏览器通知** 仅发 **1 条**，正文两行各述一条。

### 8.3 服药状态、完成率与库存扣减（评审 B-01 / B-02）

| 行为 | 是否计入当日「应服」分母 | 是否计入当日「已服/依从」分子（H-02） | 库存扣减次数 | 记录归属日 |
|------|-------------------------|----------------------------------------|----------------|-------------|
| 按时已服 | 是 | 是 | **1 次**（按该次用量） | 计划所属本地日历日 |
| 漏服（未补） | 是 | 否 | **0** | 计划所属日 |
| 漏服后补服 | 是 | **是**（计为当日依从完成） | **1 次**（仅补服这一次，**不重复** 扣原漏服那次） | **以用户操作补服的本地日历日为准** 计入该日依从；同时在原计划日上可显示「已于某日补服」链接，**不重复** 计入两日分子 |
| 跨日补服 | 原计划日分母仍计应服；补服操作日 **不增加** 当日应服分母 | 补服操作日记入分子（**与「当日是否有其他待服」无关**，见 **8.3.1**） | **1 次**，扣在补服确认时点 | **扣减库存归属补服确认日**；完成率分子归属 **补服操作日** |

- **H-02 完成率**：`当日依从完成次数 / 当日应服总次数`，其中「依从完成」= 按时已服 + 已在宽限策略内标记的补服（与上表一致）。
- **宽限期**：漏服后 **同一自然日 24:00 前** 可标记补服仍计为「针对该计划日」的依从（可配置，默认见第 10.2 节）；超过则仅记历史、是否仍允许补服扣库存由产品 **可选**（MVP 建议仍允许补服扣库存但不提升当日完成率）。
- **超期补服边界（评审补充）**：超过当日 24:00 后的补服入口 **始终开放**（用药计划 → 历史任意日均可），无天数上限限制；超期补服的库存扣减 **归属补服确认时的本地日历日**（与跨日补服一致，见上表）；**不提升** 原计划日的依从分子。

#### 8.3.1 补记入口判定（评审收口）

- **入口位置**：**用药计划** Tab → **日历或按日列表** → 用户点选 **任意历史本地日** → 展示该日应服/漏服记录 → 可对 **漏服** 条目点击 **补服**。  
- **「有无补记」**：**不依赖** 补服操作日当天是否存在其他用药计划；**任意日均可** 为历史某日补记（只要该日在系统内曾生成过应服记录且状态允许补服）。  
- **完成率**：跨日补服时 **分子仅计入补服操作日**，**原计划日不因补服操作而增加分子**（与上表一致）。

### 8.4 频次与每日清单（评审 D-03）

**MVP 支持的频次类型**（未列类型 **不支持保存**，并 Toast 提示「当前版本仅支持以下频次」）：

| 类型 | 说明 | 生成每日清单规则 |
|------|------|------------------|
| 每日 N 次 | N 为 1～6 | 按用户设定的 N 个时间点各生成 1 条待服 |
| 每 12 小时 / 每 8 小时 等 | 等间隔 | 映射为每日 N 次（N = 24/间隔小时），时间点可均分或用户自定 |
| 每日 1 次 | — | 1 条待服 |

**首版不支持**（录入时禁用或提示）：每周固定几日、隔日服、按需服、复杂 titration。**不支持时的提示策略**：表单内灰色说明 + 提交时阻断并给出示例「请拆分为多条每日计划或咨询医生简化表述」。

### 8.5 实体字段列表（简表）

- **用户档案**：慢病类型、病史、过敏史  
- **药品主数据**：名称、规格、库存单位、总库存、阈值、批次列表（实现层）  
- **用药计划**：关联药品主数据、每次剂量、剂量单位、频次类型与参数、时间点、疗程、启用状态  
- **服药记录**：计划 ID、应服日期、时段、状态（已服/漏服/补服）、操作时间戳  
- **医生 / 科室**：展示字段 + 可预约时段  
- **问诊订单**：模式、科室、医生、状态（见 Q-06）、时间、摘要  
- **处方**：药品明细、有效期、关联问诊 ID  
- **复诊事件**：时间、类型（线上/线下）、医生、科室、周期规则

## 12. 边界、异常场景与一致性约定（二次评审）

本节用于减少研发/测试 **「未写即自由发挥」** 的扯皮；与第 8 章业务规则冲突时 **以本节为补丁说明**。

### 12.1 本地数据与并发

| 场景 | 约定 |
|------|------|
| `localStorage` **接近配额**（剩余 < 200KB） | **主动预警**：在每次写入后检查估算剩余空间，低于 200KB 时 Toast **「存储空间即将用尽，建议清理旧数据」**（每次会话最多提示 1 次，不重复打扰）；用户仍可继续操作直至真正写入失败。 |
| `localStorage` **配额满** 或写入抛错 | **禁止静默失败**：Toast **「存储空间不足，部分数据未保存」**；**禁止** 在未知持久化成功时关闭编辑弹窗；关键写操作可 **重试 1 次**（间隔 300ms），仍失败则保留表单内数据并提示用户 **导出备份（二迭）或清理数据**。 |
| **数据结构升级**（版本迁移） | 应用内维护 **`schemaVersion`**，**初始版本号为 `1`**；启动时若检测到旧版本：① **缺字段** 以 **安全默认值** 填充（布尔 `false`、数字 `0`、数组 `[]`）；② **迁移函数须支持链式升级**（如 v1→v2→v3），不假设用户跳版；③ 若无法迁移（破坏性变更），**阻断进入主流程**，弹层说明 + 按钮 **「清空本地数据并升级」**（等同 U-03 清空）与 **「导出 JSON（若已实现）」**。 |
| **多标签页** 同时打开 | **不支持跨标签实时协同**；约定 **以最后写入 `localStorage` 为准**（不强制 `storage` 事件同步以避免冲突复杂度）。**可选**：监听 `storage` 事件仅用于 **提示「其他标签页已更新，是否刷新」**（MVP 可不做，但须在 README 写明「建议单标签使用」）。 |

### 12.2 时间与作弊口径

| 场景 | 约定 |
|------|------|
| 用户 **修改系统时间** | **仍信任设备本地时间** 驱动业务日界与计划切换（否则无法工程实现）；**首次检测到大步跳跃**（如一次改 ±24h 以上）时 **顶栏一次性提示**：「检测到系统时间大幅变更，用药记录以设备时间为准，请核对。」**不阻断** 使用。 |

### 12.3 库存与打卡

| 场景 | 约定 |
|------|------|
| 扣减后 **库存 < 一次用量** 或 **= 0** | **禁止** 将本次标记为 **已服/补服**（主按钮 `disabled`），Toast：**「库存不足，无法记录服药，请先调整库存或联系医生调整剂量」**。**例外**：演示开关「允许透支」**默认关闭**，不设入口；二迭若开放须单独评审。 |
| **禁止已服但分母** | 当日应服 **仍计分母**（计划仍在）；**分子不得增加** 直至库存足够后用户完成打卡。 |
| **同一时刻、同一药品多条启用计划** | 扣库存顺序：**计划 `createdAt` 升序**；用户从合并卡片打卡时 **默认按该顺序依次扣减**（或让用户选扣哪条计划，MVP 取默认序即可）。 |

### 12.4 处方、疗程与计划一致性

| 场景 | 约定 |
|------|------|
| **处方有效期早于疗程结束** | 处方过期日次日 00:00 起：该处方关联计划 **自动停用**（与 P-02 一致）；疗程未到的日期 **不再生成待服**；列表 **灰态仅可查看历史**，不可再「已服」打卡。 |
| **处方已过期，用户保留旧计划** | **仅查看** 历史打卡与台账；**不提供** 基于过期处方的「继续服药」打卡（主操作引导复诊）。 |
| **导入处方后用户改剂量** 与处方原文不一致 | 计划详情展示 **「与处方不一致」** 黄色标签（弱警示色）；**不阻断** 保存；点击标签可读 **免责声明**：「修改后的剂量由您自行确认，请以医嘱为准。」 |
| **P-06 预览差异表为空**（新旧处方药品完全一致） | 预览步骤展示文案 **「本次处方与当前用药方案一致，无需更新」**，主按钮变为 **「关闭」**，**不进入**第二步「选择范围」；不执行任何计划变更。 |

| **线下复诊（F-01）与线上面诊预约（Q）同一天、同一医生** | **不去重为两条业务数据**；首页 **H-03 只展示 1 张合并卡片**，标题 **「今日就诊安排」**，正文两行分别 **线下复诊时间** 与 **线上问诊时间**；点击进入 **日程聚合半屏** 可展开两条。 |

### 12.6 购药入库与「有库存无计划」

|------|------|
| **P-04 确认购药** 用户 **未** 勾选「同步到用药计划」 | **仅增加库存批次**；首页 **H-01 不出现该药**；**H-04** 若命中阈值仍展示；并在首页 **常驻轻提示条**（可关闭一次）：「您有新入库药品尚未加入用药计划」+ 按钮 **「去添加」** → 跳转 M-01 预填药品名。 |

### 12.7 清空数据与合规展示

| 场景 | 约定 |
